# Example: Computer Systems Curriculum — Full 20-Section Design

This is a curriculum prompt that produced excellent results. It shows how to structure a comprehensive, multi-session learning path that starts from absolute zero and builds to sophisticated understanding.

## What Made This Curriculum Design Great

- **Started from the deepest constraint** — "What is a computer?" before anything else
- **Strict dependency ordering** — never references something unexplained
- **Goal intuition per section** — each one is a quotable mental model:
  - "Computation is controlled state change"
  - "The OS exists to prevent chaos under scarcity"
  - "If you understand the filesystem, system behavior becomes predictable"
  - "The terminal is a state-manipulation environment"
  - "An API is a contract that lets one program ask another program to do something"
  - "Git is a system for tracking every change ever made"
  - "Claude Code is an AI agent with permission to read, write, and execute"
- **Explicit learner profile** — listed exactly what the learner does NOT know (OS internals, Unix, command line, Linux, programming, internet, APIs, Git, GitHub, MCPs, databases)
- **Built toward a practical end state** — not abstract knowledge, but "understand what Claude Code is doing when it runs commands, writes files, or accesses services"
- **20 sections that form a complete arc** — from physics (state machines) to AI agents acting on your behalf

## The Arc

1. Computer as physical state machine
2. Why operating systems exist (resource allocation under scarcity)
3. Filesystem as organizing idea (persistence needs structure)
4. Terminal and shell (text interfaces never disappeared)
5. Processes, users, permissions (isolation enables scalability)
6. Applications and where state lives (if it happened, it was written somewhere)
7. Programming languages (structured state transformations)
8. Command line as control surface (explicit state transitions)
9. How the internet works (computers requesting from other computers)
10. APIs (contracts between programs)
11. Authentication (how the internet knows you said yes)
12. Databases (structured storage beyond files)
13. DevOps (code from your machine to the world)
14. Version control and Git (tracking every change)
15. GitHub (code on the internet)
16. MCPs (AI tools plugging into your digital life)
17. Claude Code (AI agent with filesystem + internet access)
18. AI tools as system actors (state transformers with guardrails)
19. Big picture map (where you stand, what you chose to ignore)
20. Practice and calibration (observation-based exercises)

## Design Principles to Replicate

- Each section explicitly says what to cover AND what to skip
- "Do not cover" lists prevent going too deep
- Physical/economic constraints ground everything
- The curriculum tells a story — it's not a list of topics, it's a narrative that builds
- Visual requirement: images from the web AFTER abstraction, never decorative
- Pacing: one section at a time, pause for confirmation
